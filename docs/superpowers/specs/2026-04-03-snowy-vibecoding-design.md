# Snowy Vibecoding 基座改造设计

## 1. 背景与目标

当前 `snowy` 采用 `snowy-common + snowy-plugin + snowy-plugin-api + snowy-web-app` 的插件化 Maven 结构，适合作为快速开发平台，但不够适合后续以 vibecoding 方式持续演进。当前目标不是继续保留“插件平台”形态，而是将其收敛为一套更适合 AI 持续协作开发的 Java 后端基座。

本次改造的核心目标如下：

- 将项目收敛为“领域模块化单体”，弱化插件心智
- 默认主干只保留后台基座必须能力，降低理解成本和维护负担
- 前后端同步瘦身，避免保留无效菜单、空壳页面和死接口
- 最终作为一个新的自有项目重新初始化 Git，而不是继续沿用上游仓库历史
- 后续与官方 `snowy` 的关系采用“上游参考 + 选择性移植”，而不是直接合并更新

本次改造 **不包含 AI 模块预留**。未来 AI 能力计划通过独立 Python 服务接入，因此本轮设计只聚焦 Java 后端基座和配套前端壳层。

## 2. 目标形态

### 2.1 后端目标结构

改造后的后端保留以下五个主干模块：

- `snowy-common`
- `snowy-auth`
- `snowy-system`
- `snowy-dev`
- `snowy-boot`

目标结构特点：

- 仍然是单体项目
- 保留少量 Maven 模块，避免所有代码拍平成一个大模块
- 不再保留 `plugin-api` 这层插件式接口心智
- `boot` 只负责启动、聚合、装配，不承载业务逻辑

### 2.2 前端目标结构

前端继续基于 `snowy-admin-web`，本轮不切换到 `admin-shadcn`。前端目标是同步瘦身为一套极简后台壳。

保留主干视图域：

- `src/views/auth`
- `src/views/sys`
- `src/views/dev`
- `src/views/index`

保留必要基础层：

- `layout`
- `router`
- `store`
- `api`
- 公共组件与基础配置

## 3. 模块职责设计

### 3.1 `snowy-common`

职责：

- 通用配置
- 异常体系
- 统一返回结构
- 通用工具类
- 数据库与 MyBatis 基础能力
- 认证与权限所需的通用支撑
- 各业务模块都会依赖的基础枚举、常量、基类

约束：

- 只放真正跨领域复用的能力
- 不承载具体业务逻辑
- 原插件中的“只是被多个模块共用”的公共能力可以回收到这里

### 3.2 `snowy-auth`

职责：

- 登录
- 登出
- Token 与会话流程
- 认证相关校验
- 第三方登录相关能力

来源：

- `snowy-plugin-auth`
- `snowy-plugin-auth-api`

约束：

- 不再拆分独立 API 模块
- 对外只保留普通业务模块语义

### 3.3 `snowy-system`

职责：

- 用户
- 角色
- 菜单
- 组织
- 岗位
- 资源
- 数据权限

来源：

- `snowy-plugin-sys`
- `snowy-plugin-sys-api`

说明：

这是整个后台基座的核心业务域，后续所有业务扩展都建立在这个模块提供的组织权限主干之上。

### 3.4 `snowy-dev`

职责：

- 字典
- 参数配置
- 文件
- 日志
- 消息
- 任务

来源：

- `snowy-plugin-dev`
- `snowy-plugin-dev-api`

说明：

这里的目标不是保留 `dev` 全量能力，而是只保留作为后台底座所需的开发支撑能力。后续实施时需要继续做一次子能力筛选。

### 3.5 `snowy-boot`

职责：

- Spring Boot 启动入口
- 配置聚合
- 模块装配
- 环境入口

来源：

- `snowy-web-app`

约束：

- 不写具体业务逻辑
- 只保留启动聚合与装配职责

## 4. 默认移出主干的模块

以下模块不纳入第一阶段基座主干：

- `client`
- `mobile`
- `gen`
- `biz`

原因：

- 它们不是“最小可用后台基座”的必要部分
- 这些能力会扩大默认理解范围，不利于 vibecoding 的主路径稳定
- 传统代码生成在 AI 生成能力增强后优先级下降

处理方式：

- 后端先从默认聚合构建中移出
- 前端同步删除对应页面、菜单、路由、接口调用
- 如未来确有需要，可作为附加业务域重新接入，而不是默认常驻主干

## 5. 迁移原则

### 5.1 先重组结构，再清理细节

第一阶段不追求全量业务逻辑重写，而是优先完成：

- 模块边界重组
- Maven 结构收敛
- `plugin-api` 职责回收
- 前后端主干瘦身

结构理顺后，再进行二次清理、命名优化和细节统一。

### 5.2 去掉插件心智，但不拍平为大单体

本次改造不是简单地“保留插件目录但换个名字”，也不是把所有代码揉成一个模块，而是将项目改造成普通业务领域模块化单体：

- 模块保留
- 插件语义移除
- 跨层跳转减少
- 新增功能时改动路径更固定

### 5.3 前后端同步瘦身

后端移出的能力，前端必须同步移出：

- 菜单
- 路由
- 页面
- API 封装
- 按钮权限引用

目标是让前后端始终维持同一套“极简后台基座”，不保留无效入口和死页面。

### 5.4 默认主干只服务后台基座

第一阶段基座只服务以下能力：

- 认证
- 权限
- 用户组织体系
- 菜单权限
- 字典配置
- 文件
- 日志
- 任务

不为未来可能用到的业务场景提前预埋大量模块。

### 5.5 最终以新项目方式收尾

实施完成后：

- 删除当前仓库 `.git`
- 重新初始化 Git
- 作为新的自有项目开始维护

这意味着后续不再把该仓库视为“官方 snowy 的轻微改版”，而是视为一个基于 snowy 再造的新基座。

## 6. 前端同步改造原则

前端继续使用 `snowy-admin-web`，但同步改造成极简后台壳。

### 6.1 保留范围

保留：

- 登录认证相关页面
- 首页与基础壳层
- 系统管理相关页面
- 开发支撑相关页面

### 6.2 移出范围

移出：

- `src/views/biz`
- `src/views/client`
- `src/views/gen`
- `src/views/mobile`
- `src/views/exm`
- `src/views/front`
- `src/views/other`

### 6.3 改造原则

- 先做减法，不先做 UI 重做
- 页面删减必须同步到路由、菜单、接口层
- 前端只服务当前后端主干
- 为后续第二阶段替换前端壳保留清晰边界

## 7. 维护模式

改造完成后，项目与官方 `snowy` 的关系调整为：

**上游参考 + 选择性移植**

不采用：

- 直接 `git pull` 官方更新
- 长期保持结构兼容以便机械合并

采用：

- 跟踪官方安全修复
- 跟踪核心通用 bug 修复
- 按价值决定是否同步
- 结合当前自有结构进行移植式更新

同步优先级：

1. 安全修复
2. 认证、系统、开发支撑主干的通用问题修复
3. 通用依赖升级
4. 已被移出模块的更新默认不跟

这套维护模式的目标是：

- 保留上游价值
- 不被上游结构绑死
- 让自有基座独立演进

## 8. 成功标准

本轮改造完成后，应满足以下结果：

- 后端形成 `common + auth + system + dev + boot` 的清晰主干
- 不再存在 `plugin-api` 作为默认理解路径
- `client/mobile/gen/biz` 不再属于默认主干
- 前端与后端主干能力严格一致
- 可以正常完成最小后台基座常见主流程：
  - 登录
  - 用户/角色/菜单/组织管理
  - 字典/配置管理
  - 文件/日志/任务能力
- 项目以新仓库方式维护

## 9. 非目标

本轮不处理以下事项：

- AI 能力接入
- Python 服务设计
- 前端切换到 `admin-shadcn`
- 大规模视觉重做
- 全量业务能力保留
- 与上游保持可直接合并的结构兼容
